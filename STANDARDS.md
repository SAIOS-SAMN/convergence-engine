## 1. File Naming

**Format:** `NOUN_PHRASE.md` — the filename names the subject,
not the argument or the defense.

**Rules:**
- No `WHY_` prefix — the document's content explains why;
  the filename names the subject
- No `HOW_TO_` prefix — same reason
- Two to three words maximum. If you need more, the concept
  isn't distilled enough
- Use `UPPER_SNAKE_CASE` for documentation files
- Use `lower_snake_case` for source code files
- Numbered files (`1_`, `2_`) are for ordered sequences
  within a research category (reading order within a topic)
- The name should be readable as a table-of-contents entry:
  "this section covers ___"

**Examples:**
```
Good:  VOCABULARY_TRAINING_METHODOLOGY.md
       BIOLOGICAL_COMPREHENSION.md
       INTERNAL_TRAINING.md
       DOMAIN_BOUNDARY.md

Bad:   WHY_BIOLOGICAL_LANGUAGE.md    (defensive)
       HOW_TO_USE_THE_FRAMEWORK.md   (instructional prefix)
       A_COMPLETE_GUIDE_TO_CBE.md    (too long, article prefix)
```

## 2. Tone

**Target:** Relaxed plain academic English. The reader is a
peer — a researcher, developer, or student who is evaluating
the work on its merits.

**Rules:**
- No commanding language ("you must," "you shall," "never do")
  — describe what works, not what to obey
- No ecclesiastical language ("sacred," "holy," "divine,"
  "inviolable," "decree," "mandate") — these are design
  constraints, not doctrine
- No mythological language in active descriptions — mythology
  belongs in historical context (Layer 5 of the derivation
  tower) and is documented as methodology, not as current state
- No defensive tone ("why we had to," "the reason this was
  necessary") — state the methodology and its evidence
- Explain, don't persuade. The work either holds up to
  evaluation or it doesn't
- "We invite you to evaluate it" not "Evaluate it" — the
  reader decides, we don't instruct

**Self-test:** Read the sentence aloud. If it sounds like a
manifesto, a sermon, or a terms-of-service agreement, rewrite
it. If it sounds like a research paper or a project README,
it's correct.

## 4. Vocabulary (Documentation)

Documentation across all repositories uses plain academic
English. The same tone rules apply (Section 2).

**Additional rules for documentation:**
- "Self-consistent" not "sovereign" in descriptive prose
  (sovereign is a CCL code term, not a documentation term)
- "Design constraints" not "laws" or "rules"
- "Methodology" not "protocol" (unless describing a specific
  technical protocol)
- "Architecture" not "canon"
- "State record" not "genome" (except in historical context)
- "Entity" not "witness/elder/child" (except in historical
  context showing the Phase 1 vocabulary)

## 6. Citations

All citations are maintained in a single file:
`references/REFERENCES.md`

**Rules:**
- Numbered sequentially (1, 2, 3...)
- Each entry includes: author, year, title, one-sentence
  description of what it documents, and URL
- Documents reference by number: `[Ref 42]`
- URLs must be verified as accessible at time of publication
- The date of last verification is stated at the bottom of
  REFERENCES.md
- Do not cite papers that do not exist. If a citation cannot
  be verified by web search, it is removed

```

  DISCLAIMER.md       — AI-generated content statement
  LICENSE             — Apache 2.0
  STANDARDS.md        — this document (or link to canonical copy)
  src/                — source code (if applicable)
```

**Cross-repository references** use full GitHub URLs or
relative paths within the same repository. Never assume
directory structure outside the current repo.

## 8. Git Practices

- Main branch is the published state
- Feature branches for work in progress
- Branches are deleted after merge (no stale remote branches)
- Commit messages: one-line summary, blank line, details
- Co-Authored-By line for AI-assisted commits
- No force-push to main
- No commits containing credentials, API keys, or personal
  data
